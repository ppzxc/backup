# PRD: 백업 파이프라인 아키텍처 개선 (`backup.sh`)

## 문제 정의 (Problem Statement)

Restic 백업 파이프라인(`backup.sh`)을 유지 보수하는 시스템 관리자 및 개발자로서, 일부 핵심 도메인이 얕고 파편화된 유틸리티 함수들로 구현되어 있어 코드베이스에 아키텍처적 마찰이 발생하고 있습니다.

구체적으로:
1. **저장소 백엔드 로직 (Storage Backend Logic)**: S3 및 SFTP 백엔드 설정, 매개변수 해석, 유효성 검증, 에셋 준비 및 안내 메시지 렌더링이 백엔드당 6개씩 총 12개의 얕은 함수로 조각나 있습니다. 핵심 CLI 명령 모듈(`cmd_setting` 및 `cmd_migrate`)이 코디네이터 역할을 하며 `case` 문 분기를 통해 이 단계를 순차적으로 조합하여 실행합니다. 이로 인해 각 저장소 백엔드의 구체적인 구현 세부 정보가 CLI 명령 실행부로 누출되어 모듈의 **depth**(깊이)가 얕아지고 **locality**(응집도)를 해칩니다. 새로운 백엔드를 추가하려면 두 개의 CLI 코디네이터 코드를 여러 군데 수정해야 하는 문제가 있습니다.
2. **바이너리 설치 (Binary Installation)**: 의존성 도구(`restic`, `rclone`, `resticprofile`)를 다운로드하고 설치하는 파이프라인(임시 폴더 생성, curl 다운로드, sha256 체크섬 검증, 압축 해제, 지정 경로 설치)이 중복되어 구현되어 있습니다. 이로 인해 네트워크 프록시, 해시 검증 또는 에러 핸들링 방식을 수정할 때 일관성을 유지하기 어렵습니다.
3. **설정 해석 및 유효성 검사 (Configuration and Validation)**: 매개변수 파싱, 유효성 검사, 우선순위 해석(CLI 플래그 > 환경 변수 > 설정 파일 > 내장 기본값) 로직이 분산되어 있습니다. `cmd_setting`과 `cmd_migrate`가 각각 우선순위를 수동으로 검사하고 개별 필드를 검증하여 중복이 발생합니다.

---

## 해결 방안 (Solution)

`backup.sh`를 리팩토링하여 명확한 경계선(Seam)을 가진 깊은 모듈(Deep Module)들을 도입합니다:
1. **백엔드 어댑터 인터페이스 (Backend Adapter Interface)**: 저장소 백엔드 추상화 경계선(Backend Seam)을 도입합니다. 파편화된 함수들을 단일한 깊은 인터페이스(`backend_configure` - 환경 변수 및 알림 안내 렌더링 처리, `backend_test_connectivity` - 연결 검증)로 통합합니다. SFTP, S3 구체적 어댑터는 이 인터페이스의 세부 구현으로 숨겨져 CLI 명령부로부터 완전히 격리됩니다.
2. **바이너리 관리 모듈 (Binary Manager Module)**: 바이너리 메타데이터를 매개변수로 받아 임시 폴더 설정, curl 다운로드, SHA-256 검증, 압축 해제 처리(bz2/zip/tar), 권한 설정 및 정리를 일괄 처리하는 단일 깊은 인터페이스 `install_binary`를 정의하고 중복을 제거합니다.
3. **설정 관리 모듈 (Configuration Manager Module)**: 설정 파일(`backup.env`) 로드, 환경 변수 계층 병합, 우선순위 해석 및 유효성 검사 규칙을 하나로 통합한 순수 함수 인터페이스 `resolve_and_validate_config`를 구축하여 일관된 검증을 보장합니다.

## 사용자 스토리 (User Stories)

1. 시스템 관리자로서, 새로운 저장소 백엔드를 쉽게 추가하고 싶습니다. 이를 통해 기존 핵심 명령 로직을 깨뜨리지 않고 새로운 인프라 환경에 맞춰 백업 파이프라인을 확장할 수 있게 되기를 원합니다.
2. DevOps 엔지니어로서, 바이너리 설치 로직이 견고하게 작동하기를 원합니다. 이를 통해 설치 중 네트워크 타임아웃이나 체크섬 불일치 오류가 발생했을 때 모든 도구에서 동일하게 안정적인 에러 핸들링이 지원되기를 원합니다.
3. 개발자로서, 단일화된 설정 검증 경계선(Seam)이 있기를 원합니다. 이를 통해 새로운 CLI 옵션을 추가할 때 `setting`, `migrate`, `wizard` 전체 명령어에 걸쳐 동일한 데이터 검증 규칙이 보장되기를 원합니다.
4. 인프라 엔지니어로서, 백엔드 전용 자격 증명과 환경 변수가 각 어댑터 내부에 완전히 감춰지기를 원합니다. 이를 통해 민감한 매개변수가 CLI 명령어의 전역 범위로 누출되는 것을 방지하고 싶습니다.
5. QA 엔지니어로서, 단일한 인터페이스를 통해 백엔드 어댑터의 동작을 검증하고 싶습니다. 이를 통해 내부 헬퍼 함수가 리팩토링되더라도 테스트 코드들이 깨지지 않기를 원합니다.
6. 보안 감사원으로서, 환경 설정 파일 권한이 한 곳에서 강제되고 검증되기를 원합니다. 이를 통해 권한 설정 오류가 모든 명령어 흐름의 초기에 확실하게 검출되고 리포팅되기를 원합니다.
7. 개발자로서, 통합된 `install_binary` 인터페이스를 사용하고 싶습니다. 이를 통해 새로운 바이너리 의존성을 추가할 때 동일한 다운로드 코드를 재작성하는 대신 한 줄의 선언만으로 추가할 수 있기를 원합니다.
8. 시스템 관리자로서, `backup.sh`가 감사 보고서를 이전과 완전히 동일한 표준 출력 및 JSON 형식으로 제공하기를 원합니다. 이를 통해 기존 ISMS 컴플라이언스 로그 수집 파이프라인이나 SIEM 에이전트 수정 없이 통합이 지속되기를 원합니다.

## 구현 결정 사항 (Implementation Decisions)

### 1. 통합 백엔드 어댑터 인터페이스 (Unified Backend Adapter Interface)
* **인터페이스**: 아래의 고수준 동작을 노출하는 백엔드 경계선(Seam)을 구현합니다.
  * `backend_configure`: 백엔드 설정 매개변수를 인자로 받아 최종 렌더링된 환경 변수 설정 파일의 내용물과 권장 정책/알림 가이드를 반환합니다.
  * `backend_test_connectivity`: 백엔드의 초기 연결 여부를 사전 비행 검사(Pre-flight check)합니다.
* **어댑터**: `sftp`와 `s3`에 대한 구체적인 구현체는 이 인터페이스 뒤에 감춰집니다. CLI 명령어 핵심 로직(`cmd_setting` 및 `cmd_migrate`)은 백엔드 종류에 따라 해당 인터페이스를 동적으로 호출하므로, 백엔드 분기를 위한 개별 `case` 블록들이 CLI 코어 코드에서 제거됩니다.

### 2. 깊은 바이너리 관리 모듈 (Deep Binary Manager Module)
* **인터페이스**: 단일한 `install_binary(name, version, url, sha256, target_path, format)` 함수 인터페이스를 제공합니다.
* **동작 세부사항**: 임시 디렉터리 생성/삭제, curl 다운로드 인자 처리, 체크섬 대조 흐름을 캡슐화합니다.
  * `.bz2` 아카이브는 파이썬의 `bz2` 모듈을 이용하여 해제합니다.
  * `.zip` 아카이브는 파이썬의 `zipfile` 모듈을 이용하여 해제합니다.
  * `.tar.gz` 아카이브는 `tar` 유틸리티를 사용하여 직접 압축 해제합니다.
* 기존 `install_restic`, `install_rclone`, `install_resticprofile` 함수 내부의 명령들을 모두 단순 메타데이터 형태로 변환하여 `install_binary`로 위임합니다.

### 3. 통합 설정 관리 모듈 (Unified Configuration Manager Module)
* **인터페이스**: 단일한 `resolve_and_validate_config` 인터페이스를 도입합니다.
* **동작 세부사항**: CLI 플래그, 환경 변수, 기존 backup.env, 디폴트 값들의 우선순위 해석 및 모든 타입 유효성 검증(정수 체크, 문자열 공백 검사 등)을 단일한 순수 함수 흐름으로 처리하고, 해석 완료된 최종 상태의 맵(Map)을 반환합니다.
* 만약 검증 중 오류가 발생할 경우, 발생한 모든 오류 리스트를 일괄 수집하여 사용자에게 친절한 형식으로 요약 반환하고 조기에 종료시킵니다.

## 테스트 결정 사항 (Testing Decisions)

### 1. 테스트 경계선 (Test Seams)
* Bats 테스트 수트는 앞으로 내부 헬퍼 유틸리티 대신 새로 노출되는 통합 모듈 인터페이스를 통해 코드를 검증합니다:
  * `tests/backend_adapters.bats` 테스트는 백엔드 개별 세부 유틸리티가 아닌 `backend_configure` 및 `backend_test_connectivity`를 타깃하여 환경 변수 렌더링 값과 에러 분기를 검증합니다.
  * `tests/cmd_install.bats` 테스트는 상위 명령 테스트 시 `install_binary` 자체를 모킹/스텁 처리하고, 바이너리 매니저 기능 테스트에서는 격리하여 검증합니다.
  * 유효성 검사 관련 테스트(`tests/validators.bats`)는 `resolve_and_validate_config` 모듈을 통해 통합된 해석 결과를 기준으로 검증합니다.

### 2. 좋은 테스트의 기준
* 외부 행위 검증 중심:
  * `backend_configure`에 올바른 인자를 주었을 때 설정 파일의 exports 문자열이 올바르게 생성되는지를 검증합니다.
  * 바이너리 다운로드 중 파일이 깨졌을 때(체크섬 오류) 정상적으로 오류를 내며 실행을 종료하는지 검증합니다.
  * 설정 로더가 우선순위 규칙에 맞게 올바른 계층에서 값을 획득하는지 검증합니다.
* 내부 구현 상세(예: Zip 압축 해제 시 어떤 파이썬 내부 인자를 사용했는지 등)에 의존한 검증은 지양합니다.

### 3. 기존 테스트 자산 (Prior Art)
* 기존 `tests/backend_adapters.bats` 파일을 리팩토링하여 새 통합 백엔드 어댑터 인터페이스 테스트로 마이그레이션합니다.
* `tests/cmd_setting_s3.bats` 및 `tests/cmd_setting_sftp.bats`도 설정 관리자를 통해 검증하는 구조로 갱신합니다.

## 제외 범위 (Out of Scope)

* 이번 아키텍처 리팩토링 과정에서 새로운 저장소 백엔드 유형(예: Azure Blob, GCS)을 신규 구현하는 작업.
* `backup.sh`를 다른 언어(예: Python, Go)로 전면 포팅하는 작업.
* systemd 타이머 유닛의 설정 템플릿 구조 자체를 변경하거나, `resticprofile`이 처리하는 예약 백업 및 정기 정리(Forget/Prune)의 기본 정책 작동 방식을 변경하는 작업.

## 기타 주의 사항 (Further Notes)

* **하위 호환성**: 이번 아키텍처 개선은 생성되는 `/etc/restic/backup.env` 파일 구조나 감사 보고서 파일 저장 형식을 변경해서는 안 됩니다. 기존 모니터링 도구나 자동화 파이프라인이 정상적으로 동작하도록 출력 포맷을 일관되게 유지합니다.
* **TDD 및 린트 준수**: 새로운 세부 설계 반영 시 Bats 테스트 실패 케이스를 먼저 작성하고 구현하는 TDD 개발 프로세스를 엄격히 따릅니다. 변경 후에도 `shellcheck backup.sh` 정적 린트 경고가 0건을 유지해야 합니다.
* **버전 관리 규칙**: 작업 반영 시 `backup.sh` 내부의 `BACKUP_SCRIPT_VERSION` 변수를 반드시 상향 조정해야 합니다.
